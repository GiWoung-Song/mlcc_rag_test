# Workflow Details

이 reference는 대화 상태 관리, 입력 수집, rerun 규칙을 정리한다.

## Contents

- 기본 흐름
- 입력 수집 규칙
- 결과 제시 규칙
- rerun 규칙
- 질문 템플릿

## 기본 흐름

### 1. Reference 검증

1. `lot_id`를 확보한다.
2. `check_optimal_design(lot_id)`를 호출한다.
3. `부족인자`가 비어 있지 않으면:
   - 현재 lot으로는 시뮬레이션을 진행하지 않는다.
   - `충족인자`, `부족인자`를 보여준다.
   - 다른 `lot_id`를 요청한다.

### 2. Target 수집

reference가 유효할 때만 진행한다.

수집 원칙:

- 빠진 `targets.*`만 묻는다.
- 보통 4개 수준이므로 한 번에 묻는다.
- 기본은 `절대값`으로 받는다.

대표 예시:

- 용량
- chip thickness
- chip length
- chip width

사용자가 delta로 말하면:

- 현재 기준이 되는 값이 명확할 때만 절대값으로 변환한다.
- 기준값이 불명확하면 절대값으로 다시 확인한다.

### 3. DOE 입력값 수집

수집 원칙:

- 빠진 `params.*`만 묻는다.
- 약 10개 수준이면 한 번에 답하기 어렵지 않도록 2개 묶음으로 나눌 수 있다.
- 현재 payload 기준은 `절대값`이다.
- tool schema에 field 설명이 있으면 그 설명에 맞춰 질문한다.

params.* 값 형식은 실행 유형에 따라 다르다:

- **초기 실행**: 사용자에게 각 params 항목의 범위(예: ref 대비 ±5%)와 포인트 수(예: 21포인트)를 수집한 뒤, ref lot 값을 중심으로 등간격 절대값 리스트를 생성한다.
- **재실행**: 선택한 후보의 설계값을 그대로 가져와 각 params 항목을 `[단일값]` 리스트로 만든다. 사용자가 변경을 지시한 항목만 해당 값으로 교체한다.

사용자가 범위를 지정하지 않으면 기본값을 제안한다: `ref 대비 ±3%, 11포인트`. 사용자에게 확인받은 뒤 리스트를 생성한다.

### 4. Payload 조립

Payload를 만들 때는 다음을 사용한다.

- 현재 `lot_id`
- 최신 `targets.*`
- 최신 `params.*`

조립 규칙:

- 이미 알고 있는 값은 유지한다.
- 사용자가 이번 턴에 수정한 값만 덮어쓴다.
- 빈 값이나 추정값은 넣지 않는다.

### 5. 시뮬레이션 실행

1. `optimal_design`을 호출한다.
2. top 5 후보를 결과 순서대로 번호 `1~5`로 붙인다.
3. 내부적으로 각 후보의 전체 설계값을 보존한다.

### 6. 공정검사표준 검증

시뮬레이션 결과를 사용자에게 보여주기 **전에** 공정검사표준 부합 여부를 확인한다.

1. top 5 후보의 핵심 설계값 항목(Sheet T, Electrode W, Margin L, Margin W, Cover T, 전극수 등)을 대상으로 `search_rag`를 호출한다.
   - 쿼리 예: `"공정검사표준 Sheet 두께 허용범위"`, `"검사표준 전극 폭 Margin 상한 하한"`
   - 유사 항목은 묶어서 검색해도 된다.
2. 검색된 표준에서 각 항목의 상한/하한을 추출한다.
3. top 5 각 후보의 해당 값이 표준 범위 안인지 비교한다.
4. 결과 테이블에 공정검사표준 부합 여부 열을 추가한다:
   - ✅ 범위 내
   - ⚠️ 초과 (어느 값이 어떤 한계를 넘었는지 명시)
   - ❓ 미확인 (해당 항목의 표준 정보를 찾지 못한 경우)
5. 표준 초과 항목이 있는 후보는 요약에서 trade-off로 명시한다.

## 입력 수집 규칙

- 이미 확보된 `lot_id`를 다시 묻지 않는다.
- 같은 run 내에서 이미 확보된 `targets.*`, `params.*`를 다시 묻지 않는다.
- 사용자가 `그건 그대로`라고 말하면 직전 값을 유지한다.
- 사용자가 일부 값만 수정하면 나머지는 유지한다.

## 결과 제시 규칙

각 후보마다 최소한 아래를 보여준다.

- 후보 번호
- 핵심 설계값
- 예측 성능
- target과의 차이
- 추천 이유 또는 trade-off

가능하면 다음 형식으로 요약한다.

1. 후보 1: 가장 target 근접
2. 후보 2: 두께 여유가 큼
3. 후보 3: 용량은 좋지만 width가 불리함

내부 상태에는 반드시 아래를 유지한다고 가정한다.

- 최신 `lot_id`
- 최신 `targets`
- 최신 `params`
- 최신 `top_candidates`
- 각 candidate의 전체 설계값

## Rerun 규칙

사용자가 예를 들어 아래처럼 말하면:

- `3번째 설계값에서 Sheet T만 5.2로 바꿔서 다시`
- `2번 후보에서 width만 0.15 늘려서 다시`

다음 순서로 처리한다.

1. 최신 top 5 결과에서 지정한 후보 번호를 찾는다.
2. 해당 후보의 전체 설계값을 base로 복사한다.
3. 사용자가 바꾸라고 한 field만 override한다.
4. 새 payload를 만든다. 이때 `params.*`는 각 항목을 **단일 값 리스트** `[value]`로 구성한다. 초기 실행처럼 다중 포인트 리스트를 만들지 않는다.
5. `optimal_design`을 다시 호출한다.
6. 결과가 나오면 다시 공정검사표준 검증(Step 6)을 수행한 뒤 사용자에게 제시한다.

주의:

- 최신 top 5 결과가 현재 문맥에 없으면 어떤 후보를 말하는지 먼저 확인한다.
- 후보 번호는 항상 최신 top 5 기준이다.
- field 이름이 실제 candidate design 값에 없으면 추측하지 말고 정확한 필드명을 다시 확인한다.

## 질문 템플릿

### Reference 부족 시

- `이 reference LOT은 바로 시뮬레이션에 쓰기 어렵습니다. 충족인자: ... / 부족인자: ... 입니다. 다른 lot_id를 주시면 다시 검증하겠습니다.`

### Target 수집 시

- `시뮬레이션 target을 위해 아래 4개 절대값이 필요합니다: 용량, chip thickness, chip length, chip width. 현재 아는 값만 제외하고 알려주세요.`

### Params 수집 시

- `DOE 입력값 중 아직 비어 있는 항목은 다음과 같습니다: ... 절대값 기준으로 알려주세요.`

### 다건 후보 후속 질문

- `top 5 후보를 계산했습니다. 이 중 어느 후보를 base로 다시 흔들지 알려주세요. 예: 3번째 후보에서 Sheet T만 5.2로 변경`
